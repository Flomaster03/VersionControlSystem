Описание
Система контроля версий — это программное обеспечение, которое может отслеживать изменения, внесенные в программу. Существует несколько популярных систем контроля версий (например, Git, SVN или Mercurial). У каждого из них есть свои плюсы и минусы. Их объединяет одна общая идея. Система контроля версий запоминает, кто изменил файл, когда это было сделано и почему . Это также позволяет вам вернуться к предыдущим версиям.

В этом проекте вам необходимо реализовать простую систему контроля версий. Это будет ваш собственный Git. Кстати, если вы хотите научиться работать с Git, загляните в Git How To .

Взгляните на команды, которые вам нужно будет реализовать в этом проекте:

configустанавливает или выводит имя автора коммита;
--helpраспечатывает страницу справки;
addдобавляет файл в список отслеживаемых файлов или выводит этот список;
logпоказывает все коммиты;
commitсохраняет изменения файла и имя автора;
checkoutпозволяет переключаться между коммитами и восстанавливать предыдущее состояние файла.
На этом этапе ваша программа должна иметь возможность принимать один аргумент и, в зависимости от аргумента, печатать справочную информацию.

На этом этапе ваша программа должна:

Возьмите один аргумент как команду.
Если аргумент отсутствует или есть --help, распечатайте всю страницу справки.
Если команда существует, программа должна вывести описание команды.

В отличие от многих других проектов, программа не работает в бесконечном цикле. Программа начинается с аргументов, которые передаются в основную функцию и должны что-то сделать, затем снова начинается с нуля с другими аргументами и должна делать что-то еще.

Описание
На этом этапе вы реализуете две ключевые команды. commitпозволит пользователю сохранять изменения файлов; logпозволит просматривать историю коммитов. Цель состоит в том, чтобы смоделировать базовую функциональность VCS, например Git, где изменения файлов сохраняются как «коммиты» и можно просматривать историю этих коммитов.

Команда commitпозволяет пользователю сохранять изменения, внесенные в файл. Каждый коммит должен иметь уникальный идентификатор коммита. Этот идентификатор может быть сгенерирован с использованием криптографического хеша (например, SHA-1) на основе содержимого файла или временной метки. Коммит следует создавать только в том случае, если в файлах есть изменения с момента последнего коммита. Это можно определить, сравнив хеш текущего состояния с хешем последней зафиксированной версии.

Если файл/файлы изменились, система:

Создайте новый каталог внутри, vcs/commits/названный по уникальному идентификатору коммита;

Скопируйте текущую версию всех отслеживаемых файлов в этот новый каталог;

Запишите идентификатор фиксации и сообщение о фиксации в vcs/log.txtфайл;

Если ни один из файлов не изменился с момента последнего коммита, система выведет сообщение типа «Нечего коммитить», и новый коммит создан не будет.

Команда logпозволяет пользователю просматривать историю коммитов. История коммитов будет сохранена в vcs/log.txtфайле, где каждая запись содержит:

Идентификатор фиксации;

Сообщение о фиксации;

После logвыполнения команды система отобразит историю коммитов в обратном порядке (сначала будет показан самый последний коммит).

Git может показаться довольно сложным. Если вы хотите узнать больше, посмотрите видеообъяснение от GitLab.
